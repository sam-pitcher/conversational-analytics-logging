LookML repository designed to analyze and monitor logs generated by the **Google Conversational Analytics API**.

By pairing the backend logging (capturing Looker and BigQuery agent responses) with this LookML model, you can gain deep insights into agent performance, user intent, and common failure points like missing data or schema gaps.

# Conversational Analytics Monitoring (LookML)

This repository serves as the semantic layer for tracking interaction logs from AI agents. It doesn't just display logs; it uses **BigQuery ML (Gemini)** to automatically classify the success of conversations and identify specific data gaps in real-time.

## 🚀 Core Features

* **Log Parsing:** Automatically flattens complex JSON payloads from the Conversational Analytics API into queryable dimensions (SQL, Chart Configs, Looker Model/Explore context).
* **AI-Driven Classification:** Uses `ML.GENERATE_TEXT` to categorize responses into `SUCCESS`, `MISSING_DATA`, `DISAMBIGUATION`, etc.
* **Missing Field Extraction:** A chained ML model specifically identifies which column or measure a user asked for that wasn't available in the current Looker Explore.
* **Performance Tracking:** Built-in measures for conversation duration, message counts, and interaction latency.

---

## 🏗️ Architecture & Views

### 1. Interaction Logs (`interaction_logs`)

The foundation of the model. It handles the heavy lifting of extracting JSON values from the `raw_json` field provided by your backend.

* **Key Dimensions:** `user_query`, `generated_sql`, `looker_explore`, and `chart_title`.
* **Latency Metrics:** Tracks `total_duration` to monitor how fast the agent is responding to users.

### 2. Status Classification (`status`)

An incremental Persistent Derived Table (PDT) that sends the log content to the `gemini-2.5-flash` model in BigQuery.

* **Purpose:** To understand *why* a user didn't get an answer.
* **Efficiency:** Uses `incrementcondition` on the timestamp to ensure we only classify new logs, saving on API costs.

### 3. Missing Field Analysis (`status_missing_field`)

A specialized view that triggers only when a log is flagged as `MISSING_FIELD`. It asks the LLM to extract the exact name of the missing attribute so you can prioritize your LookML development backlog.

## 🔑 IAM & BigQuery Setup

To allow Looker to execute the ML functions and access the Gemini models, the Service Account used for your **BigQuery Connection** needs specific permissions.

### 1. Required Roles

The Looker Service Account requires the following roles in the Google Cloud Project:

* **BigQuery Data Editor:** To create and update the PDTs (Status and Missing Field views).
* **BigQuery Job User:** To run the SQL queries.
* **Vertex AI User:** Required to call the `ML.GENERATE_TEXT` function.

### 2. Enabling the Model

The SQL in this repo references a specific model:
`MODEL sampitcher-playground.conversation_logs.gemini_2_5_flash`

1. Ensure the **Vertex AI API** is enabled in your GCP project.
2. Create the remote model connection in BigQuery (via `CREATE MODEL`) before deploying this LookML.
3. Grant the Service Account access to the specific dataset where the ML model resides.

---

## 🛠️ Installation

1. Connect this GitHub repo to your Looker Project.
2. Ensure your BigQuery connection name matches `default_bigquery_connection` (or update it in the model file).
3. Update the `sql_table_name` in the `interaction_logs` view to point to your backend's raw log table.
4. Deploy to production to trigger the initial PDT build.
